## **CPU vs GPU**

![Untitled](Untitled.png)

> CPU는 속도가 빠른 8대의 비행기로 짐을 실어 나르는 것에 비유 할 수 있고, GPU는 속도는 느리지만 1000대의 기차로 짐을 실어 나르는 것에 비유 할 수 있습니다.
> 

CPU는 산출연산 처리장치(ALU)와 명령어를 해석, 실행하는 컨트롤 유닛(CU), 각종 데이터를 담아두는 캐시로 나뉩니다. GPU는 캐시 메모리 비중이 크지 않고, 대신 ALU 개수가 많습니다.  ALU의 개수가 GPU에 비해 적은 CPU는 CPU는 명령어가 입력된 순서대로 데이터를 처리하는 직렬 처리 방식에 특화된 구조를 가지고 있고, 반대로 GPU는 병령 처리 방식을 가지고 있어 처리해야 할 데이터의 성격에 따라 CPU와 GPU를 잘 선택해야합니다.

셰이더란 뭘까요?

대학에 들어오기 전, 고등학교 개발을 이어 갈때도 항상 이런건 셰이더로, 이건 셰이더로 처리를... 이라는 말을 듣으면서 셰이더를 직접 만드는 경험을 하기까지 많이 들어왔지만, 한 번도 ‘셰이더’라는 단어의 뜻에 대해 생각해 본 적이 없었습니다.  (생각해 보자구!)

셰이더란 화면에 출력할 픽셀의 위치와 색상을 계산하는 함수입니다. 함수란 입력 값을 받아서 처리하고, 그 결과를 출력하는 것 입니다. 이 개념을 셰이더에 대입 한다면 3D 모델링의 메쉬(mesh) 데이터들을 받아 → 3D 공간의 모델링을 픽셀로 변환해 → 픽셀들의 최종 색상들을 출력하게 됩니다. 

![Untitled](Untitled%203.png)

3D 모델링들은 폴리곤을 이루고 있는 버텍스들과 normal, UV data들을 갖고 있습니다. 이런 3D Model들은 메테리얼이 없이는 화면에 렌더링 될 수 없으며, 이런 메테리얼들은 Texture와 Shader 속성이 필요합니다. 

![출처 : [https://m.blog.naver.com/camelkill/220516812987](https://blog.naver.com/camelkill/220516812987)](Untitled%204.png)

출처 : [https://m.blog.naver.com/camelkill/220516812987](https://blog.naver.com/camelkill/220516812987)

Shader는 보통 HLSL과 GLSL 언어를 사용해 만들어지는데(외에 CG 같은 다양한 언어가 있긴 하지만 생략), GLSL(OpenGL Shading Language, OpenGl 셰이딩 언어)는 C언어를 기초로 한 상위 레벨 셰이딩 언어입니다. HLSL과 유사하지만 하드웨어에 의존한 언어를 사용하지 않고, 그래픽스 파이프라인으르 직접 제어할 수 있도록 개발 되었습니다. 운영체제 간의 호환성이 좋고, 그래픽 카드의 처리정치에 최적화된 코드를 만들 수 있습니다. 반대로 HLSL(High Level Shader Language)는 마이크로소프트 타이렉트3D API에 사용되는 언어로, OpenGL API에서 표준으로 사용되는 셰이딩 언어인 GLSL와 유사한 특징을 보입니다. 고급 언어이기에 가독성이 좋고 관리도 쉽습니다. C와 C++과 유사한 문법을 가지고 있기에 배우기 쉬운 편입니다. 

## Direct3D 11 Pipeline

![[https://docs.microsoft.com/ko-kr/windows/uwp/graphics-concepts/graphics-pipeline](https://docs.microsoft.com/ko-kr/windows/uwp/graphics-concepts/graphics-pipeline)](Untitled%205.png)

[https://docs.microsoft.com/ko-kr/windows/uwp/graphics-concepts/graphics-pipeline](https://docs.microsoft.com/ko-kr/windows/uwp/graphics-concepts/graphics-pipeline)

Direct3D 그래픽 파이프 라인은 실시간 그래픽을 생성하는데 설계된 파이프 라인입니다. 모서리가 둥근 사각형 블록(밑에 노란색으로 표시된 문장)은 HLSL 프로그래밍 언어를 사용해 프로그래밍 할 수 있습니다. 가장 일반적으로 프로그래밍 하는 단계는 VS(vertex shader)와 PS(Pixel Shader)입니다. 

IA : 원시 데이터(point, line, triangle)를 읽어서 다른 스테이지에서 사용될 데이터(Line list, triangle strips)으로 어셈블하는 역할을 합니다. 

VS : 버텍스들을 처리하는 단계. transformation(위치 변환), sknning, morphig, vertex lighting 등의 가공을 거쳐서 버텍스 정보로 출력합니다.

HS : 폴리곤을 어떻게 얼마나 분할할지, 스킵 여부를 처리합니다.

TS : 폴리곤을 더 잘게 나누는 작업을 합니다.(quad → triangle), 실제 분할된 버텍스 정보를 DS에 전달합니다. 

DS : 잘게 나누어진 point의 vertex position을 계산하는 단계입니다.

GS : 이전 단계의 버텍스 정보를 인풋으로 받아 가공하여 새로운 폴리곤(혹은 선이나 점)을 구성하여 출력합니다.

RS : 뷰 컬링(View Culling), 3D 폴리곤을 2D 픽셀로 변환, 보간(Interpolation), Pixel Shader용 데이터들을 준비합니다.

PS : 픽셀 라이팅이나 post-processing 같은 셰이딩을 처리하는 단계입니다. 파이프 라인 중 가장 무거운 스테이지 입니다. 

OM : 최종적으로 출력되는 픽셀의 색을 결정합니다. Depth & Stencil을 고려하여 렌더링을 결정하고, blend 합니다.

Z Buffer는 오브젝트 하나가 그려질 때마다 Ztest(테스트)를 거쳐 Zwrite(업데이트)를 하는 과정을 거쳐 제작된 텍스처를 말합니다. 조금 더 간단하게 말하자면 픽셀이 얼마나 멀리 있는지 거리 정보를 기록하는 것이라고 표현 할 수도 있겠네요. 제일 먼게 1이고 제일 가까운 거리를 0으로 기록합니다.

![Untitled](Untitled%2014.png)

이런 상황이 있다고 할때,

![Untitled](Untitled%2015.png)

Z-buffer에 거리를 기록해 놓음으로써 물체의 간격, 거리를 판단해 화면으로 잘 표현 할 수 있습니다. 

![Untitled](Untitled%2016.png)

여기 카메라 세팅에 near와 far 값이 0.3, 1000 값이 들어가 있는 카메라가 있습니다. 

이 카메라는 0.3m부터 시작해 1000m까지 렌더링 하겠다는 뜻 입니다. 즉 1km 멀리 있어도 보인다는 것입니다. Z buffer는 near부터 far까지 가까운 곳은 0, 먼곳은 1로 기록하기 때문에 near와 far 값이 크게 차이 날 수록 정확도는 떨어지게 됩니다. 

![Untitled](Untitled%2017.png)

 반대로 10m로 화각을 설정했다면 위 이미지처럼 같이 멀리 있는 것은 안보이지만, 화면 안에 있는 물체 중 가까이 있는 물체들을 더 정확하게 거리 연산을 할 수 있습니다.

![Untitled](Untitled%2018.png)

카메라에 보이는 정보를 리니어 z buffer로 저장하게 된다면 빨간색 체크 범위처럼 균등하게 기록하겠지만 우리는 멀리 있는 정보보다 가까이 있는 물체의 뎁스가 중요하기 때문에 가까이 보이는 것을 더 정밀하게 기록해야 합니다. 그래서 파란색 체크와 같은 간격으로 z buffer를 체크 할 수 있도록 해야 합니다.

## Zbuffer의 Z-Test와 Z-Write

우리는 Z buffer의 처리 과정을 test와 write로 나누어 생각해 볼 것입니다. 

화면에 그림을 그리기 전 Z값을 보고, 앞에 있는지 없는지 체크하는 것을 Test(Read)라고 하고, 내 앞에 있는게 없어지면(앞에 아무것도 없으면) 그려주는 것을 Write라고 하겠습니다.

그렇다면 4가지 상황이 있네요!

**Z test = on, Z write = on**

읽기도 하고 쓰기도 합니다. 이게 기본입니다. 물체 앞에 무엇이 있는지 체크하고, 앞에 아무것도 없으면 그려주니 아래처럼 문제 없이 잘 출력 될 것 같습니다.

![Untitled](Untitled%2015.png)

**Z test = off, Z write = off**

읽지도 않고 쓰지도 않습니다. 

이 경우 내 앞에 어떤 물체들이 있어도 무조건 뒤에 그려지기 때문에 항상 뒤에 들어가는 스카이 박스 같은 것에 유용합니다. 

**Z test = off, Z write = on**

앞에 뭐가 있던 무조건 자기가 그릴 타이밍에 그려버립니다. 

UI 같이 화면 상단에 항상 존재 해야 하는 것들에 유용하겠네요!

**Z test = on, Z write = off**

읽기는 하는데, 쓰지는 않습니다. 이건 어떻게 활용 할 수 있을까요?

![Untitled](Untitled%2019.png)

![Untitled](Untitled%2020.png)

왼쪽은 불투명, 오른쪽은 반투명 오브젝트입니다. 이를 유니티에선 Opaque와 Transparent라고 부릅니다. 불투명 오브젝트는 우리가 생각하기에 서로 겹쳐지는 부분은 블렌딩 될 것이고, 물체의 뒤로 간다면 물체에 가려 보여지지 않을 것입니다. 하지만 더 가까운 위치에 있다고 반투명 오브젝트가 먼저 그려지고, 나중에 멀리 있는 불투명 오브젝트가 그려진다면 불투명 오브젝트는 반투명 오브젝트와 겹친 부분에서 자신의 픽셀을 그리면 안된다고 판단해 다음과 같은 상황이 발생할 수도 있습니다. 

![누가.. 앞에 있는거지..?](Untitled%2021.png)

누가.. 앞에 있는거지..?

그래서 반투명 오브젝트는 ZBuffer를 그리지 않기 때문에 ZWrite를 꺼줘야 합니다. 

Forward Renderer에서만 가능한 반투명 렌더링은 ZWrite를 끄고 렌더링합니다. ZWrite를 켜면 폴리곤 형태로 가로막는다.
